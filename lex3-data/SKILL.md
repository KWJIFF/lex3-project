# Lex3 数据沉淀

## 功能

案件数据收集、清洗、结构化、分析、应用。

## 工作流

数据收集 → 数据清洗 → 数据结构化 → 数据分析 → 数据应用 → 反馈优化

---

## 节点详解

### Node 1: 数据收集（15动作）

```
1.1.1 制定数据收集计划：
  - 明确收集目标
  - 列出数据来源
  - 制定时间表
1.1.2 从案件提取：
  - lex3-project案件完成后提取数据
  - 案件类型/案由/金额/结果
1.1.3 从律师收集：
  - 律师办案数据
  - 律师专长/收费/评价
1.1.4 从公开渠道获取：
  - 裁判文书网
  - 执法公示
  - 行业报告
1.1.5 从内容提取：
  - 阅读量/转化数据
  - 热门选题
1.1.6 整理数据：
  - 分类存放
  - 格式统一
1.1.7 初步筛选：
  - 去除重复
  - 去除无效
1.1.8 分类：
  - 案件数据
  - 律师数据
  - 内容数据
  - 客户数据
1.1.9 检查完整性：
  - 是否有缺失
  - 是否需补充
1.1.10 标记缺失：
  - 记录缺失字段
  - 计划补充
1.1.11 导入数据仓库
1.1.12 定期更新：
  - 每日/每周/每月
1.1.13 建立收集机制：
  - 自动/手动
1.1.14 监控进度：
  - 收集完成度
  - 数据质量
1.1.15 处理问题：
  - 识别并解决收集问题
```

### Node 2: 数据清洗（12动作）

```
2.2.1 检查重复性：
  - 识别重复记录
2.2.2 删除重复：
  - 保留最新/最完整记录
2.2.3 检查准确性：
  - 核实数据真实性
2.2.4 纠正错误：
  - 修正错误数据
  - 标准化格式
2.2.5 标准化：
  - 统一格式（日期/金额等）
  - 统一命名规范
2.2.6 脱敏处理：
  - 去除敏感信息
  - 保护隐私
2.2.7 补充缺失：
  - 通过其他渠道补充
  - 标记为"待补充"
2.2.8 验证质量：
  - 检查数据准确率
  - 生成质量报告
2.2.9 生成清洗报告：
  - 清洗前后对比
  - 问题记录
2.2.10 存入数据库：
  - 导入目标数据库
  - 验证导入成功
2.2.11 更新数据字典：
  - 记录字段含义
  - 记录变更历史
2.2.12 记录清洗过程：
  - 归档日志
```

### Node 3: 数据结构化（10动作）

```
3.3.1 设计存储结构：
  - 数据库表结构
  - 字段定义
3.3.2 分类存储：
  - 按类型分区
  - 按时间分区
3.3.3 标签标注：
  - 添加业务标签
  - 添加质量标签
3.3.4 建立索引：
  - 常用查询字段
  - 加速查询
3.3.5 构建关联：
  - 案件-律师关联
  - 律师-客户关联
  - 内容-转化关联
3.3.6 设计接口：
  - API设计
  - 权限设计
3.3.7 测试性能：
  - 查询速度
  - 并发能力
3.3.8 编写文档：
  - 数据说明
  - 使用指南
3.3.9 培训使用：
  - 内部培训
  - 问题解答
3.3.10 持续优化：
  - 根据使用反馈优化结构
```

### Node 4: 数据分析（15动作）

```
4.4.1 确定分析目标：
  - 业务问题
  - 决策需求
4.4.2 选择分析方法：
  - 描述性分析
  - 诊断性分析
  - 预测性分析
4.4.3 案件类型分析：
  - 各类案件占比
  - 趋势变化
4.4.4 地域分析：
  - 案件地域分布
  - 地域特征
4.4.5 金额分析：
  - 涉案金额分布
  - 平均金额
4.4.6 结果分析：
  - 判决结果分布
  - 缓刑/实刑比例
4.4.7 趋势分析：
  - 案件数量趋势
  - 类型变化趋势
4.4.8 对比分析：
  - 不同律师对比
  - 不同策略对比
4.4.9 识别异常：
  - 识别异常数据
  - 分析异常原因
4.4.10 生成图表：
  - 可视化展示
  - 便于理解
4.4.11 撰写分析报告：
  - 发现 insights
  - 给出建议
4.4.12 分享结果：
  - 内部分享
  - 决策参考
4.4.13 收集反馈：
  - 报告质量
  - 改进建议
4.4.14 优化方法：
  - 改进分析方法
  - 丰富分析维度
4.4.15 归档结果：
  - 保存报告
  - 记录方法
```

### Node 5: 数据应用（10动作）

```
5.5.1 为内容提供素材：
  - 典型案例脱敏
  - 数据支撑观点
5.5.2 为技术提供需求：
  - 分析工具需求
  - 自动化需求
5.5.3 为服务提供支撑：
  - 律师推荐依据
  - 报价参考
5.5.4 为决策提供依据：
  - 阶段升级判断
  - 资源配置
5.5.5 建立API：
  - 数据查询接口
  - 分析结果接口
5.5.6 设计产品：
  - 数据产品规划
  - 商业化方案
5.5.7 推广产品：
  - 内部使用
  - 外部合作
5.5.8 收集反馈：
  - 使用体验
  - 功能需求
5.5.9 优化产品：
  - 根据反馈优化
  - 增加新功能
5.5.10 探索变现：
  - 数据服务收费
  - 分析报告售卖
```

### Node 6: 反馈优化（8动作）

```
6.6.1 收集反馈：
  - 用户反馈
  - 使用问题
6.6.2 识别问题：
  - 数据质量问题
  - 分析质量问题
6.6.3 分析效果：
  - 数据应用效果
  - 决策支持效果
6.6.4 优化收集：
  - 改进收集流程
  - 扩大收集范围
6.6.5 优化清洗：
  - 提高清洗效率
  - 改进清洗质量
6.6.6 优化结构：
  - 改进存储结构
  - 优化查询性能
6.6.7 优化分析：
  - 改进分析方法
  - 增加分析维度
6.6.8 更新规范：
  - 制定新规范
  - 优化现有规范
```

## 阶段配置

| 阶段 | 数据量 | 启用 |
|------|--------|------|
| 1-2 | <20 | 暂停 |
| 3 | 20-50 | 启动 |
| 4 | 50-100 | 活跃 |
| 5+ | 100+ | 核心 |
