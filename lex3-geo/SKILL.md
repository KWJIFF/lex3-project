# Lex3 GEO优化系统

## 功能定位

专门负责内容在AI平台（ChatGPT/Claude/Perplexity/豆包/文言一心等）的引用优化，提升品牌曝光和案源获取。

---

## 核心概念

### GEO vs SEO

| 维度 | SEO | GEO |
|------|-----|-----|
| 目标 | 搜索引擎排名 | AI答案引用 |
| 平台 | Google/Baidu | ChatGPT/Claude/Perplexity/豆包 |
| 核心 | 关键词+外链 | 知识结构+权威性 |
| 流量 | 点击流量 | AI推荐流量 |
| 用户 | 搜索关键词 | 提问完整问题 |

### 为什么律师行业需要GEO

1. **高净值案源**：AI推荐的用户信任AI判断，转化率高
2. **被动获客**：一次优化，长期被动获客
3. **信任背书**：被AI引用=权威认可
4. **竞争窗口**：现在入局=占据先发优势

---

## 工作流

```
问题研究 → 内容结构化 → 语料库构建 → 第三方布局 → 长尾问题覆盖 
→ llms.txt部署 → 多平台分发 → 效果追踪 → 持续优化
```

---

## 节点详解

### Node 1: 问题研究（18动作）

```
1.1.1 确定目标法律领域：
  - 刑事辩护
  - 婚姻家事
  - 劳动纠纷
  - 合同纠纷
  - 知识产权

1.1.2 确定目标AI平台：
  - ChatGPT
  - Claude
  - Perplexity
  - 百度文言一心
  - 字节豆包

1.1.3 搜索平台用户问题（真实长尾问题）：
  - "上海离婚律师推荐"
  - "离婚时房子怎么分"
  - "老公出轨了房子归谁"
  - "婚后买的房一定是夫妻共同财产吗"
  - "请律师离婚要多少钱"

1.1.4 收集完整问题：
  - 问题主体
  - 具体情况
  - 地域
  - 诉求

1.1.5 问题分类：
  - 信息型（什么是X）
  - 行动型（如何做X）
  - 风险型（X安全吗/违法吗）
  - 比较型（XvsY选哪个）
  - 案例型（类似情况怎么判）

1.1.6 提取问题中的关键实体：
  - 地域（上海/北京/深圳）
  - 案由（离婚/继承/工伤）
  - 财产（房产/车辆/存款）
  - 人物（原告/被告/第三人）

1.1.7 分析问题复杂度：
  - 简单问题（可直接回答）
  - 中等问题（需补充条件）
  - 复杂问题（需多角度分析）

1.1.8 搜索现有AI回答：
  - 在各AI平台搜索同类问题
  - 记录AI回答的内容
  - 分析AI引用的来源

1.1.9 识别引用机会：
  - AI回答不完整
  - AI回答过时
  - AI回答角度单一
  - AI缺少案例支撑

1.1.10 竞争分析：
  - 哪些律所/律师被引用
  - 引用的内容是什么
  - 我们的差异化在哪里

1.1.11 关键词提取：
  - 核心关键词
  - 长尾关键词
  - 地域关键词

1.1.12 问题优先级排序：
  - 高价值（高搜索+低竞争）
  - 机会型（有需求+少回答）
  - 防守型（对手已布局）

1.1.13 建立问题库：
  - 问题ID
  - 问题内容
  - 搜索量估计
  - 优先级
  - 目标答案要点

1.1.14 定期更新问题库：
  - 每周新增问题
  - 跟踪问题趋势

1.1.15 输出问题研究报告
```

### Node 2: 内容结构化（20动作）

```
2.2.1 理解GEO内容vs公众号内容：
  - 公众号：故事开头→分析→结尾
  - GEO：问题→法条→分析→结论→注意事项

2.2.2 结构化内容模板（FAQ型）：
  - 问题是什么（直接回答）
  - 法律依据（引用法条）
  - 深度分析（多角度）
  - 典型案例（脱敏）
  - 注意事项（实操建议）

2.2.3 开头优化（AI首句引用）：
  - 首句直接回答问题
  - 30字内给出核心答案
  - 包含关键词
  - 禁止先讲故事

2.2.4 标题层级优化：
  - H1：主标题（包含关键词）
  - H2：章节标题（问题导向）
  - H3：子节标题（细分问题）
  - 每个标题必须能独立回答一个问题

2.2.5 要点列表优化：
  - 使用bullet points
  - 每点不超过20字
  - 关键信息前置
  - 避免长段落

2.2.6 表格优化：
  - 数据对比用表格
  - 流程用表格
  - 表格有明确标题
  - 行列标签清晰

2.2.7 法条引用规范：
  - 引用原文（不能断章取义）
  - 标注出处（法律法规名称+条款）
  - 区分主文本和司法解释

2.2.8 案例引用规范：
  - 案情脱敏处理
  - 标注案号
  - 提取裁判要点
  - 说明参考价值

2.2.9 数据引用规范：
  - 标注数据来源
  - 标注统计时间
  - 说明统计口径

2.2.10 避免AI踩坑：
  - 禁止绝对化表述（"一定"/"必须"）
  - 禁止情感色彩（"非常遗憾"/"极其过分"）
  - 禁止模糊表述（"可能"/"大概"）
  - 禁止过时信息（标注更新时间）

2.2.11 语义单元分割：
  - 每段不超过3句话
  - 每段必须有核心句
  - 段落长度均衡
  - 段落可独立引用

2.2.12 结构化数据标记：
  - Article schema
  - FAQ schema
  - HowTo schema
  - LegalService schema

2.2.13 Author Authority：
  - 作者姓名、资质、背景
  - 律所名称、资质
  - 专业领域说明

2.2.14 更新标注：
  - 首次发布时间
  - 最后更新时间
  - 更新内容说明

2.2.15 可验证性：
  - 所有事实可查证
  - 提供查证路径
  - 注明核实方式

2.2.16 内部链接：
  - 链接到相关文章
  - 链接到专业页面
  - 形成内容网络

2.2.17 外部引用：
  - 链接到权威来源
  - 链接到官方文件
  - 增强可信度

2.2.18 自检清单：
  - [ ] 首句直接回答问题
  - [ ] 有层级标题
  - [ ] 有要点列表
  - [ ] 有数据表格
  - [ ] 有FAQ结构
  - [ ] 法条有引用来源
  - [ ] 案例有脱敏处理
  - [ ] 无绝对化表述
  - [ ] 有Author信息
  - [ ] 有更新时间
```

### Node 6: llms.txt部署（12动作）

```
6.6.1 理解llms.txt：
  - AI专用读取文件
  - 网站摘要
  - VIP绿色通道

6.6.2 创建llms.txt文件：
  - 文件位置：根目录
  - 文件格式：Markdown
  - 内容极简化

6.6.3 llms.txt内容结构：
  ```
  # 网站名称
  [简短描述]
  
  ## 核心业务
  - 业务1
  - 业务2
  - 业务3
  
  ## 律师团队
  - 律师1：[专业领域]
  - 律师2：[专业领域]
  
  ## 核心内容
  - [内容1]: [链接]
  - [内容2]: [链接]
  
  ## 联系方式
  - 电话：[号码]
  - 邮箱：[邮箱]
  - 地址：[地址]
  ```

6.6.4 部署到主站：
  - 上传llms.txt
  - 验证可访问
  - 验证格式正确

6.6.5 配置爬虫协议：
  - robots.txt允许
  - sitemap包含

6.6.6 多站点部署：
  - 主站
  - 博客
  - 法律工具站

6.6.7 定期更新：
  - 业务变化时更新
  - 律师变动时更新
  - 内容更新时更新

6.6.8 验证AI读取：
  - 测试AI是否能读取
  - 测试AI引用效果

6.6.9 替代方案llm.txt：
  - 某些平台用llm.txt

6.6.10 扩展属性（optional）：

  - 是否被AI读取

```
